# RisingPunk Case Study

## Project Type

React Native mobile strategy application

## Summary

RisingPunk is a React Native mobile application built for Android and iOS. The project combines mobile app development, API-backed functionality, database-driven systems, staging and production environments, and deployment infrastructure.

This case study focuses on the technical and product work involved in building a mobile product with full stack support behind it.

## Problem

The project needed more than a simple mobile interface. It required a working mobile application supported by back-end systems, database-backed functionality, API communication, and deployment environments.

The key challenge was building a mobile product where the front end, application state, API layer, database, and deployment setup could work together as one system.

## Key Features and Systems

## React Native Mobile Application

The mobile app was built with React Native to support Android and iOS from a shared codebase.

This required thinking through:

- Mobile screen structure
- Navigation
- State management
- API communication
- Mobile performance
- Platform-specific behavior
- Product iteration across Android and iOS

## API-Backed Functionality

RisingPunk uses API-backed functionality to connect the mobile application to back-end systems.

The API layer supports communication between the app, database, and server-side business logic.

Important considerations included:

- Structuring API routes
- Connecting mobile actions to server logic
- Handling asynchronous data
- Managing request and response behavior
- Keeping client and server responsibilities separated

## Database-Driven Systems

The application relies on database-backed functionality to support persistent product behavior.

Database-backed systems are important because mobile applications often need to maintain user data, product state, progression, or other persistent information across sessions.

Important considerations included:

- Modeling data clearly
- Connecting API routes to database actions
- Keeping state consistent between client and server
- Supporting future product complexity

## Staging and Production Environments

The project used separate staging and production environments.

This helped separate testing and development work from production-facing behavior.

Important environment concerns included:

- Production website routing
- Staging API environment
- Production API environment
- API subdomain routing
- Environment-specific configuration
- Safer deployment workflows

## Docker Compose Deployment

Docker Compose was used as part of the deployment workflow to organize application services and support more reliable deployment behavior.

This helped with:

- Service organization
- Environment consistency
- Local and deployment setup structure
- Managing application services together
- Reducing deployment confusion

## Cloudflare, SSL, and Certificates

Cloudflare and SSL/certificate setup were part of the deployment and production readiness work.

This involved production concerns such as:

- Domain routing
- API subdomain routing
- SSL termination
- Certificate handling
- Secure HTTPS access
- Production traffic configuration

## Technical Challenges

## Building Across Mobile, API, Database, and Deployment Layers

One of the biggest challenges was that RisingPunk was not only a front-end mobile app. It required coordination across the mobile client, API layer, database, and deployment infrastructure.

This meant decisions in one layer affected the others.

For example:

- Mobile screens needed API data
- API routes needed database models
- Database structure affected app behavior
- Environment setup affected testing and release workflows
- Deployment decisions affected API access and production behavior

## Managing Product Complexity

Mobile strategy applications can become complex because they often involve many moving pieces.

Important complexity areas included:

- App state
- User data
- Mobile navigation
- API communication
- Database-backed behavior
- Product systems that may evolve over time
- Testing behavior across development, staging, and production

## Separating Staging and Production

Creating separate staging and production environments helped reduce risk and made it easier to test changes before exposing them to production behavior.

Important considerations included:

- Keeping environment variables organized
- Separating API endpoints
- Avoiding accidental production changes
- Testing deployment changes safely
- Maintaining consistent configuration across environments
